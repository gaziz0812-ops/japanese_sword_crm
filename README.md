# Japanese Sword CRM / ERP

Система для учета товаров, поставок, склада, заказов, продаж, возвратов и отчетов для продаж через Telegram.

Проект вырос из реального процесса: товары публикуются в Telegram-канале, клиенты пишут в личные сообщения, заказы нужно фиксировать, остатки контролировать, себестоимость считать по поставкам, а прибыль видеть не "на глаз", а в системе.

Главная идея проекта: не просто сделать каталог, а собрать легкую CRM/ERP для небольшого продавца в Telegram, где заказ, продажа, склад, себестоимость и аналитика связаны между собой.

## Что умеет проект

- Вести каталог товаров с ценой, описанием, фото и статусом активности.
- Хранить производителей и поставки.
- Принимать товар через обычные и ручные поставки.
- Создавать складские партии товара.
- Считать остатки не вручную, а по движениям склада.
- Списывать себестоимость продаж по FIFO.
- Создавать заказы из Telegram Mini App.
- Привязывать заказы к Telegram-пользователю.
- Проводить продажу на основании оплаченного заказа.
- Запрещать продажу сверх остатка.
- Оформлять возвраты и списания.
- Резервировать товар и снимать резерв.
- Отправлять Telegram-уведомления администратору и клиенту.
- Показывать аналитику в Django Admin.

## Архитектура Django-приложений

### `manufacturers`

Производители товаров.

Хранит название, контакты и заметки по производителю.

### `products`

Каталог товаров.

Хранит артикул, название, производителя, описание, цену, фото и активность товара. Остаток товара не хранится отдельным полем, а считается через складские движения.

### `supplies`

Поставки товара.

Есть два сценария:

- обычная поставка: курс, доставка, комиссия, вес, расчет себестоимости;
- ручная поставка: себестоимость указывается вручную, без лишних полей про юани и доставку.

Поставка создает партии товара и движение склада типа "приход".

### `stock`

Складская логика.

Основные сущности:

- `StockMovement` - журнал движений склада;
- `StockBatch` - партия товара для FIFO;
- `StockWriteOff` - списание товара;
- `StockReservation` - резерв товара;
- allocation-модели - связь операций с конкретными партиями.

Склад работает через события: приход, продажа, возврат, списание, резерв и снятие резерва.

### `orders`

Заказы клиентов.

`Order` - заявка клиента или вручную созданный заказ администратора.

`OrderItem` - позиция товара внутри заказа.

Заказ сам по себе не списывает склад. Склад меняется только после проведения продажи.

### `sales`

Продажи и возвраты.

`Sale` - факт оплаченной продажи. Создается на основании заказа и списывает товар со склада по FIFO.

`SaleReturn` - возврат по продаже. Может вернуть товар на склад или списать его.

Продажи не являются корзиной и не создаются вручную как основной сценарий. Продажа - это исторический факт после оплаты.

### `users`

Кастомная модель пользователя.

Хранит роль, Telegram ID и Telegram username. Нужна для связи заказов с Telegram-пользователями и будущего развития ролей.

### `reports`

Отчеты и аналитика в Django Admin.

Показывает:

- выручку;
- возвраты;
- чистую выручку;
- прибыль;
- средний чек;
- топ товаров;
- заказы по статусам;
- складские показатели;
- товары с низким остатком.

### `telegram_bot`

Telegram-бот.

Открывает Telegram Mini App и используется для уведомлений.

## Главный бизнес-сценарий

1. Клиент открывает Telegram Mini App.
2. Смотрит каталог товаров.
3. Добавляет товары в корзину.
4. Оформляет заказ.
5. Django создает `Order` и `OrderItem`.
6. Telegram-пользователь подтягивается из initData.
7. Администратор получает уведомление в Telegram.
8. Администратор общается с клиентом и подтверждает оплату.
9. Заказ переводится в статус "Оплачен".
10. Администратор запускает действие "Провести продажу".
11. Система создает `Sale` по позициям заказа.
12. Продажа списывает партии товара по FIFO.
13. Создаются движения склада.
14. В заказ добавляется трек-номер и дата отправки.
15. Клиент получает уведомление об отправке.

## Складская логика

Остаток товара считается не из поля `Product.quantity`, а из движений склада.

Это важно, потому что остаток должен быть результатом истории:

- приход увеличивает остаток;
- продажа уменьшает остаток;
- возврат может вернуть товар;
- списание окончательно убирает товар;
- резерв временно убирает товар из доступного остатка;
- снятие резерва возвращает товар в доступный остаток.

Партии товара нужны для FIFO. Если один и тот же товар пришел в разных поставках с разной себестоимостью, продажа должна списывать сначала старую партию.

## Защиты от ошибок

В проекте заложены проверки, чтобы администратор не ломал бизнес-логику вручную:

- нельзя продать больше, чем есть на складе;
- нельзя вернуть больше, чем было продано;
- нельзя провести продажу по неоплаченному заказу;
- нельзя повторно провести продажу по одному заказу;
- нельзя менять позиции заказа после проведения продажи;
- нельзя вручную поставить статус "Продажа проведена" без действия проведения продажи;
- нельзя откатить отправленный заказ назад;
- нельзя удалить или изменить критичные складские операции, которые уже повлияли на остатки.

## API

Публичный API используется для каталога, Telegram Mini App и создания заказов.

Основные маршруты:

- `GET /api/products/` - список товаров;
- `GET /api/products/<id>/` - карточка товара;
- `POST /api/orders/` - создание заказа;
- `GET /api/orders/my/` - заказы текущего Telegram-пользователя.

Публичный API не должен отдавать внутреннюю информацию:

- контакты производителей;
- себестоимость;
- партии товара;
- прибыль;
- внутренние комментарии;
- служебные складские движения.

## Telegram-интеграция

Telegram Mini App передает в backend данные пользователя через initData.

Backend проверяет initData, чтобы не доверять данным, которые клиент мог подменить вручную.

После проверки система может:

- создать или обновить пользователя;
- привязать заказ к Telegram ID;
- сохранить Telegram username;
- показать клиенту историю его заказов.

Telegram-бот используется для:

- открытия Mini App;
- уведомления администратора о новом заказе;
- уведомления клиента о создании заказа;
- отправки клиенту трек-номера.

## Отчеты

Отчеты находятся в Django Admin.

Сводка продаж показывает:

- выручку;
- возвраты;
- чистую выручку;
- прибыль;
- средний чек;
- количество продаж;
- количество проданных единиц;
- топ товаров.

Складской блок показывает:

- активные товары;
- товары в наличии;
- товары без остатка;
- товары с низким остатком;
- общее количество единиц на складе;
- розничную стоимость склада.

### 1. Backend

Перейти в папку backend-проекта:

```powershell
cd "C:\Users\gaga1\Documents\Проектный проект\japanese_sword"
```

Активировать виртуальное окружение:


```powershell

```

Применить миграции:

```powershell
python manage.py migrate
```

Запустить Django:

```powershell
python manage.py runserver 8000
```

Админка:

```text
http://127.0.0.1:8000/admin/
```

### 2. Frontend

В отдельном терминале:

```powershell
cd "C:\Users\gaga1\Documents\Проектный проект\frontend"
npm run dev
```

Обычно Vite запускается на:

```text
http://127.0.0.1:5173/
```

### 3. Telegram bot

В отдельном терминале из папки `japanese_sword`:

```powershell
python -m telegram_bot.main
```

### 4. ngrok

Для тестирования Telegram Mini App нужен HTTPS.

Обычно поднимаются два туннеля:

```powershell
ngrok http 5173
```

и:

```powershell
ngrok http 8000
```

URL фронтенда указывается в `TELEGRAM_WEB_APP_URL`.

Backend ngrok URL нужен, если Telegram Mini App должен обращаться не к локальному `127.0.0.1`, а к публичному HTTPS-адресу.

## Ручной чек-лист проверки перед сдачей

### Каталог

- Создать производителя.
- Создать товар.
- Добавить цену, описание и фото.
- Убедиться, что товар виден в API и Mini App.

### Поставка

- Создать обычную поставку.
- Добавить позицию товара.
- Проверить, что создалась партия товара.
- Проверить, что создалось движение склада типа "приход".
- Проверить, что остаток товара увеличился.

### Ручная поставка

- Создать ручную поставку.
- Указать себестоимость и примечание.
- Проверить, что в форме нет лишних полей про юани, вес и доставку.
- Проверить, что партия и складское движение создались корректно.

### Заказ

- Оформить заказ через Mini App.
- Проверить, что заказ появился в админке.
- Проверить, что подтянулся Telegram-пользователь.
- Проверить, что администратору пришло уведомление.

### Продажа

- Перевести заказ в статус "Оплачен".
- Указать дату оплаты.
- Запустить действие "Провести продажу".
- Проверить, что создались продажи по позициям заказа.
- Проверить, что списание прошло по FIFO.
- Проверить, что остаток товара уменьшился.

### Отправка

- Указать трек-номер.
- Указать дату отправки.
- Перевести заказ в статус "Отправлен".
- Проверить, что клиент получил уведомление с трек-номером.

### Возврат

- Создать возврат по продаже.
- Проверить, что нельзя вернуть больше проданного.
- Проверить вариант возврата на склад.
- Проверить вариант списания возврата.

### Отчеты

- Открыть сводку продаж в админке.
- Проверить выручку, возвраты, чистую выручку и прибыль.
- Проверить складской блок.
- Проверить товары с низким остатком.

## Что еще нужно сделать для более взрослой версии

### PostgreSQL

Сейчас используется SQLite. Для production и более серьезной демонстрации лучше перейти на PostgreSQL.

Причина: PostgreSQL ближе к реальным backend-проектам, лучше подходит для продакшена и указан в исходном ТЗ.

### Docker

Следующий инфраструктурный шаг - Docker Compose:

- backend;
- PostgreSQL;
- frontend;
- возможно отдельный сервис бота.

Это упростит запуск проекта на другом компьютере.

### Тесты

Нужно добавить точечные тесты на бизнес-логику:

- нельзя продать больше остатка;
- продажа списывает FIFO;
- возврат не может быть больше продажи;
- заказ создает продажи только после оплаты;
- публичный API не отдает внутренние поля.

### Логирование

Минимально стоит логировать:

- создание заказа;
- проведение продажи;
- ошибки Telegram-уведомлений;
- критичные складские операции.

### Redis и Celery

На текущем этапе Redis и Celery не обязательны.

Они понадобятся, если:

- Telegram-уведомления нужно отправлять в фоне;
- появится тяжелый импорт Excel;
- появятся регулярные отчеты;
- HTTP-запросы нельзя будет задерживать внешними API.

### Мониторинг

Prometheus, Grafana и Sentry пока не нужны для MVP.

Их можно добавить позже, если проект будет разворачиваться для постоянной эксплуатации и появится необходимость следить за ошибками, скоростью ответов и состоянием сервера.
