# Acceptance Checklist

Чек-лист ручной приемки проекта перед сдачей или демонстрацией.

Цель: пройти основной бизнес-сценарий и убедиться, что каталог, поставки, склад, заказы, продажи, возвраты, Telegram-интеграция и отчеты работают согласованно.

## 1. Подготовка окружения

- [ ] Backend запускается без ошибок.

```powershell
cd "C:\Users\gaga1\Documents\Проектный проект\japanese_sword"
python manage.py runserver 8000
```

- [ ] Frontend запускается без ошибок.

```powershell
cd "C:\Users\gaga1\Documents\Проектный проект\frontend"
npm run dev
```

- [ ] Telegram bot запускается без ошибок.

```powershell
cd "C:\Users\gaga1\Documents\Проектный проект\japanese_sword"
python -m telegram_bot.main
```

- [ ] ngrok-туннель для frontend работает.
- [ ] ngrok-туннель для backend работает, если Mini App ходит в публичный backend URL.
- [ ] В `.env` указаны актуальные значения:
  - [ ] `DJANGO_SECRET_KEY`
  - [ ] `DJANGO_DEBUG`
  - [ ] `DJANGO_ALLOWED_HOSTS`
  - [ ] `TELEGRAM_BOT_TOKEN`
  - [ ] `TELEGRAM_ADMIN_CHAT_ID`
  - [ ] `TELEGRAM_WEB_APP_URL`

## 2. Проверка Django

- [ ] Django system check проходит без ошибок.

```powershell
python manage.py check
```

- [ ] Миграции применены.

```powershell
python manage.py migrate
```

- [ ] Админка открывается:

```text
http://127.0.0.1:8000/admin/
```

- [ ] Можно войти под суперпользователем.

## 3. Каталог товаров

- [ ] Создать производителя.
- [ ] Создать товар.
- [ ] Указать:
  - [ ] артикул;
  - [ ] название;
  - [ ] производителя;
  - [ ] описание;
  - [ ] цену;
  - [ ] фото;
  - [ ] статус "Активен".
- [ ] Проверить, что товар отображается в списке товаров админки.
- [ ] Проверить, что у товара отображается остаток.
- [ ] Проверить, что товар виден в публичном API:

```text
GET /api/products/
```

- [ ] Проверить, что товар виден в Telegram Mini App.

## 4. Обычная поставка

- [ ] Создать обычную поставку.
- [ ] Указать:
  - [ ] дату поставки;
  - [ ] курс юаня;
  - [ ] комиссию карго;
  - [ ] общую доставку;
  - [ ] вес с упаковкой;
  - [ ] комментарий при необходимости.
- [ ] Добавить позицию поставки.
- [ ] Указать:
  - [ ] товар;
  - [ ] количество;
  - [ ] цену в юанях;
  - [ ] доставку по Китаю;
  - [ ] вес 1 шт.
- [ ] Проверить, что рассчитались:
  - [ ] цена товара в рублях;
  - [ ] доставка позиции;
  - [ ] себестоимость 1 шт.
- [ ] Проверить, что создалась партия товара.
- [ ] Проверить, что создалось движение склада типа "Приход".
- [ ] Проверить, что остаток товара увеличился.

## 5. Ручная поставка

- [ ] Создать ручную поставку.
- [ ] Убедиться, что в форме нет лишних полей про юани, вес, доставку и карго.
- [ ] Добавить ручную позицию поставки.
- [ ] Указать:
  - [ ] товар;
  - [ ] количество;
  - [ ] себестоимость 1 шт.;
  - [ ] примечание к себестоимости при необходимости.
- [ ] Проверить, что создалась партия товара.
- [ ] Проверить, что создалось движение склада типа "Приход".
- [ ] Проверить, что остаток товара увеличился.

## 6. Публичный API каталога

- [ ] Список товаров открывается:

```text
GET /api/products/
```

- [ ] Карточка товара открывается:

```text
GET /api/products/<id>/
```

- [ ] Поиск работает по названию или артикулу.
- [ ] Фильтр наличия работает.
- [ ] Фильтр цены работает.
- [ ] Публичный API не отдает внутренние данные:
  - [ ] себестоимость;
  - [ ] прибыль;
  - [ ] контакты производителя;
  - [ ] внутренние комментарии;
  - [ ] партии товара;
  - [ ] служебные складские движения.

## 7. Telegram Mini App

- [ ] Mini App открывается из Telegram-бота.
- [ ] Каталог товаров загружается.
- [ ] Карточка товара открывается.
- [ ] Фильтры и сортировка работают.
- [ ] Товар добавляется в корзину.
- [ ] Корзина доступна без прокрутки всей ленты товаров.
- [ ] Маска телефона работает.
- [ ] Заказ можно оформить.
- [ ] После оформления заказа появляется понятное сообщение пользователю.

## 8. Создание заказа

- [ ] После оформления заказа через Mini App в админке появляется новый `Order`.
- [ ] В заказе заполнены:
  - [ ] имя клиента;
  - [ ] телефон;
  - [ ] Telegram username;
  - [ ] покупатель/User, если Telegram initData передан корректно;
  - [ ] позиции заказа;
  - [ ] сумма заказа.
- [ ] Цена позиции заказа зафиксирована на момент оформления.
- [ ] Изменение текущей цены товара после создания заказа не меняет старый заказ.
- [ ] Администратору приходит Telegram-уведомление о новом заказе.
- [ ] Клиенту приходит Telegram-уведомление о создании заказа, если он доступен для отправки сообщений.

## 9. Работа с заказом до оплаты

- [ ] Заказ можно редактировать до проведения продажи.
- [ ] Можно добавить позицию.
- [ ] Можно изменить количество.
- [ ] Можно указать скидку.
- [ ] Сумма заказа пересчитывается.
- [ ] Склад при создании и редактировании заказа не списывается.

## 10. Оплата и проведение продажи

- [ ] Нельзя провести продажу, если заказ не в статусе "Оплачен".
- [ ] Нельзя поставить статус "Оплачен" без даты оплаты.
- [ ] После указания даты оплаты заказ можно перевести в статус "Оплачен".
- [ ] Действие "Провести продажу" доступно в админке заказов.
- [ ] После действия "Провести продажу":
  - [ ] создается `Sale` по каждой позиции заказа;
  - [ ] каждая продажа связана с заказом;
  - [ ] каждая продажа связана с позицией заказа;
  - [ ] склад списывается;
  - [ ] создается движение склада типа "Продажа";
  - [ ] партия списывается по FIFO;
  - [ ] себестоимость продажи рассчитана;
  - [ ] прибыль рассчитана;
  - [ ] заказ переходит в статус "Продажа проведена".
- [ ] Повторное проведение продажи по этому же заказу не создает дубли.
- [ ] После проведения продажи позиции заказа нельзя менять.
- [ ] После проведения продажи заказ нельзя откатить в рабочие статусы.

## 11. FIFO

- [ ] Создать две партии одного товара с разной себестоимостью.
- [ ] Провести продажу, которая затрагивает обе партии.
- [ ] Проверить, что сначала списалась старая партия.
- [ ] Проверить `SaleCostAllocation`.
- [ ] Проверить, что прибыль считается от фактической себестоимости списанных партий.

## 12. Отправка заказа

- [ ] Нельзя перевести заказ в статус "Отправлен" без трек-номера.
- [ ] Нельзя перевести заказ в статус "Отправлен" без даты отправки.
- [ ] После указания трек-номера и даты отправки статус "Отправлен" сохраняется.
- [ ] Клиенту отправляется Telegram-уведомление с трек-номером.
- [ ] Отправленный заказ нельзя откатить назад.

## 13. Возврат

- [ ] Возврат создается по заказу и продаже.
- [ ] Из выбранного заказа можно выбрать нужную продажу/позицию.
- [ ] Нельзя вернуть больше, чем было продано.
- [ ] Можно оформить частичный возврат.
- [ ] Можно вернуть товар на продажу.
- [ ] При возврате на продажу:
  - [ ] создается движение склада типа "Возврат";
  - [ ] товар возвращается в партии, из которых был продан;
  - [ ] остаток увеличивается.
- [ ] Можно списать возвращенный товар.
- [ ] При списании возврата:
  - [ ] создается движение склада типа "Возврат";
  - [ ] создается движение склада типа "Списание";
  - [ ] итоговый доступный остаток не увеличивается.

## 14. Списание

- [ ] Можно создать списание товара.
- [ ] Нельзя списать больше доступного остатка.
- [ ] После списания создается движение склада типа "Списание".
- [ ] Партии уменьшаются по FIFO.
- [ ] Списание нельзя удалить.
- [ ] Списание нельзя изменить после создания.

## 15. Резерв

- [ ] Можно создать резерв товара.
- [ ] Нельзя зарезервировать больше доступного остатка.
- [ ] После резерва создается движение склада типа "Резерв".
- [ ] Доступный остаток уменьшается.
- [ ] Резерв нельзя удалить напрямую.
- [ ] Резерв можно снять.
- [ ] После снятия резерва создается движение склада типа "Снятие резерва".
- [ ] Остаток возвращается.

## 16. Движения склада

- [ ] В журнале движений склада видны:
  - [ ] приходы;
  - [ ] продажи;
  - [ ] возвраты;
  - [ ] списания;
  - [ ] резервы;
  - [ ] снятия резерва.
- [ ] В движении виден товар.
- [ ] В движении виден тип движения.
- [ ] В движении виден тип источника.
- [ ] В движении виден ID источника.
- [ ] Критичные движения нельзя свободно редактировать руками.

## 17. Отчеты и аналитика

- [ ] В админке доступна сводка продаж.
- [ ] Отображается выручка.
- [ ] Отображаются возвраты.
- [ ] Отображается чистая выручка.
- [ ] Отображается прибыль.
- [ ] Отображается средний чек.
- [ ] Отображается количество продаж.
- [ ] Отображается количество проданных единиц.
- [ ] Отображается топ товаров.
- [ ] Отображаются заказы по статусам.
- [ ] Отображается складской блок:
  - [ ] активные товары;
  - [ ] товары в наличии;
  - [ ] товары без остатка;
  - [ ] товары с низким остатком;
  - [ ] общее количество единиц на складе;
  - [ ] розничная стоимость склада.

## 18. Проверка защит от ошибок

- [ ] Нельзя продать больше остатка.
- [ ] Нельзя вернуть больше проданного.
- [ ] Нельзя провести продажу дважды.
- [ ] Нельзя провести продажу по неоплаченному заказу.
- [ ] Нельзя изменить позиции заказа после проведения продажи.
- [ ] Нельзя поставить статус "Продажа проведена" вручную без проведения продажи.
- [ ] Нельзя отправить заказ без трек-номера и даты отправки.
- [ ] Нельзя удалить проведенную продажу через админку.
- [ ] Нельзя удалить возврат через админку, если это запрещено бизнес-логикой.
- [ ] Нельзя сломать остатки ручным редактированием складских журналов.

## 19. Что показать на демонстрации

Минимальный сценарий для презентации:

1. Открыть Telegram Mini App.
2. Показать каталог и карточку товара.
3. Добавить товар в корзину.
4. Оформить заказ.
5. Показать заказ в админке.
6. Показать, что подтянулся Telegram-пользователь.
7. Поставить статус "Оплачен".
8. Провести продажу.
9. Показать продажи и FIFO-списание.
10. Показать движение склада.
11. Добавить трек-номер и отправить заказ.
12. Показать уведомление клиенту.
13. Открыть отчеты.
14. Показать выручку, прибыль и складской блок.

## 20. Критерии готовности MVP

- [ ] Клиент может оформить заказ через Telegram Mini App.
- [ ] Администратор видит заказ в Django Admin.
- [ ] Заказ можно провести в продажу только после оплаты.
- [ ] Продажа списывает склад по FIFO.
- [ ] Остатки считаются из складских движений.
- [ ] Возвраты и списания корректно меняют склад.
- [ ] Telegram-уведомления работают.
- [ ] Отчеты показывают ключевые показатели.
- [ ] README и ARCHITECTURE описывают проект для внешнего читателя.
